Action Permission System

A self-hosted permission layer for AI bots. The bot asks before it acts; policies auto-allow the routine, humans approve the rest, and everything — every attempt, every decision, every rule change — lands in an audit trail.

01

Overview

The system sits between a bot and the outside world as a decision service: before performing any action — a database query, a POST to an API, sending an email — the bot submits a structured description of the action and waits for a verdict. The service replies with one of three answers:

allow/ deny/ pending — a human must decide

Default posture is fail-closed: until someone writes a policy saying otherwise, every action requires human approval. Policies carve out the routine ("reads are fine") so humans only see what matters. The service never speaks SQL or HTTP itself — it judges action descriptions — which is what keeps it generic across any action type.

Approvals are single-use: one approval unlocks exactly one execution of one request, then it is consumed. If the bot wants something to become repeatable, it proposes a policy — and bot-proposed policies themselves require human approval. That recursion, capped by a depth limit, replaces "standing grants" entirely.

03

Components

Go service (single binary)

  • REST API — bots check actions and poll verdicts; humans decide; both manage policies. Versioned under /v1.
  • Policy engine — evaluates a request against active policies in priority order. Policy selection is indexed, not scanned: because wildcards are trailing-only, an action type like db.admin.drop can only ever match the candidate patterns db.admin.drop, db.admin.*, db.*, * — depth + 1 hash/btree lookups regardless of total policy count. Payload matching then goes through a pluggable Matcher interface: exact and regex in v1; llm ("ask a model to judge") is a planned second implementation, fail-closed and rationale-logged.
  • Approval manager — owns the request state machine, enforces first-decision-wins atomically, expires stale requests.
  • Audit logger — every state change writes an append-only event; nothing in the code path updates or deletes events.
  • Storage — PostgreSQL via a thin store layer (pgx + versioned migrations); Svelte build embedded with embed.FS.

Svelte dashboard

  • Approval queue — pending requests, oldest/most-urgent first. Shows the raw payload, not just the bot's summary (see summary-spoofing, open questions).
  • Policy editor — humans create, version, and disable policies directly (no queue for their own rules); every change is audit-logged with its author.
  • Policy review — bot-proposed policies rendered human-readably, with a diff against what the bot could already do.
  • Audit browser — filterable timeline: what the bot tried, which policy decided, who said yes.
  • Kill switches — disable a misbehaving bot (deny-all) or suspend all auto-allow system-wide, each one click and audit-logged.
  • Login — individual accounts; every decision is attributed to a person.

SDK (bot-side helper)

  • v1 ships a Go client; the HTTP contract is deliberately plain so any language can integrate with two endpoints.
  • Check(action) submits and blocks: instant verdict if a policy matched, otherwise polls until a human decides or the request expires. To the bot, a gated action just looks slow. An optional TTL parameter requests a longer window than the 15-minute default.
  • Sends an idempotency_key so a retried submit can't create duplicate pending requests.
  • The API reserves a webhook registration endpoint so "notify the bot instead of polling" can be added without breaking the contract.

Core API surface (sketch)

  • POST /v1/actions — submit an action for judgment
  • GET /v1/actions/:id — poll status / verdict
  • POST /v1/actions/:id/executed — bot reports outcome
  • POST /v1/actions/:id/decision — human approves/denies
  • GET /v1/policies — list policies with status + version lineage
  • POST /v1/policies — create a policy. Session-authenticated humans activate immediately (author = approver, depth 0); API-key bots enter the approval queue. Both paths are audit-logged.
  • POST /v1/policies/:id/decision — human approves/rejects a bot proposal
  • POST /v1/policies/:id/versions — edit = create a successor version (immutable lineage)
  • POST /v1/policies/:id/disable — retire a policy and its bot-made descendants
  • POST /v1/bots/:id/disable — per-bot kill switch: deny-all, instantly
  • PUT /v1/system/auto-allow — global kill switch: suspend/restore all auto-allow
  • GET /v1/audit — filterable event stream
  • POST /v1/webhooks — reserved for v1.x

04

How a decision is made

An action request carries a type (e.g. db.query, http.post, email.send — free-form, defined by the integrator) and a payload of parameters. The engine evaluates all active policies whose action_type_pattern matches, runs each policy's matcher against the payload, and combines results by strict precedence:

denyoverrides require_approvaloverrides allow·  no match → require_approval
  • Deny always wins — order never decides the outcome. The engine sorts matching policies by priority (ascending, so 10 runs before 100), evaluates their matchers, and combines all fired effects by precedence. Unlike "first match wins" systems, no allow can shadow a deny by being earlier in the list; priority only fixes which same-effect policy gets credited in the audit trail, and lets evaluation stop early once a deny fires.
  • Silence is a question, not a yes. Anything no policy speaks to goes to a human — this is the "default all actions to user approval" posture.
  • Errors fail closed. A broken regex, a store error, a future LLM-matcher timeout — all resolve to require_approval, never to allow.
  • Kill switches sit above the policies. A disabled bot gets deny on everything before any policy runs. When the global auto_allow_suspended flag is on, every allow verdict is downgraded to require_approval (denies stay denies) — one click puts a human back in front of every action without touching a single policy.
  • Every evaluation is recorded with the matched policy and its version, so the audit trail can always answer "why was this allowed?"
Example policy set for a database bot:
db.query + regex ^\s*SELECT → allow  ·  db.query + regex users|payments → require_approval  ·  db.query + regex DROP|TRUNCATE → deny. A SELECT * FROM payments matches both an allow and a require_approval rule — precedence sends it to a human. Exactly right.

05

Data model

Seven tables in PostgreSQL. Conventions: primary keys are ULIDs stored as TEXT (sortable by creation time); timestamps are TIMESTAMPTZ (always UTC); structured payloads are JSONB, validated in the app layer and queryable in SQL. PK = primary key, FK = foreign key.

users

the humans who log in and decide
ColumnTypeConstraintsMeaning
idTEXTPK (ULID)Stable identifier referenced by every decision.
nameTEXTNOT NULLDisplay name shown in the queue and audit trail.
usernameTEXTUNIQUE NOT NULLLogin identifier — no email required to run the system.
password_hashTEXTNOT NULLbcrypt hash in crypt() format, computed by Postgres's pgcrypto extension — so it can be written and verified with plain SQL (see below), while staying a proper salted hash.
roleTEXTNOT NULL DEFAULT 'approver'Reserved for future tiers (e.g. only admins approve policies). v1 treats everyone as equal.
created_atTIMESTAMPTZNOT NULLWhen the account was created.
disabled_atTIMESTAMPTZNULLSoft delete — history must keep pointing at real people, so users are never hard-deleted.
Password recovery is a SQL one-liner, by design. Self-hosted means no "forgot password" email flow to lean on. Because hashes are produced by pgcrypto's crypt(), an admin with database access can reset any account directly:
UPDATE users SET password_hash = crypt('new-password', gen_salt('bf')) WHERE username = 'alice';
Login verification is the same function: password_hash = crypt(entered_password, password_hash). Still salted bcrypt — the escape hatch costs nothing in hash strength. Requires CREATE EXTENSION pgcrypto; in the initial migration.

06

Key flows

A — Auto-allowed action allow

  1. BotSubmits db.query with payload {"sql": "SELECT id FROM orders …"}.
  2. ServicePolicy "db-reads-auto" (regex ^\s*SELECT) matches with effect allow; no deny or require_approval rule fires. Request stored as auto_allowed, audit event written with policy id + version.
  3. BotReceives allow in the same round-trip, executes the query, reports executed.

B — Human approval pending → approved

  1. BotSubmits http.post to a payments API. No policy allows it — verdict require_approval, status pending, expires_at set from the requested TTL (default 15 min; the bot may ask for longer up front). SDK starts polling.
  2. HumanAlice sees it in the queue — full payload, not just the bot's summary — and clicks Approve.
  3. ServiceAtomic transition to approved, decided_by = alice; audit event written.
  4. BotNext poll returns approved; bot performs the POST exactly once and reports executed — the approval is now consumed. Doing it again means a brand-new request.

C — Nobody answers pending → expired

  1. BotSubmits an action; goes pending.
  2. Service15 minutes pass. The expiry sweeper marks the request expired; audit event written by actor system.
  3. BotPoll returns expired — treated exactly like a deny. Fail closed.

D — Two approvers race first decision wins

  1. HumanAlice and Bob both have request #42 open.
  2. ServiceAlice's Approve lands first: UPDATE action_requests SET status='approved', decided_by='alice' WHERE id=42 AND status='pending' — one row affected.
  3. ServiceBob's click runs the same statement a moment later — zero rows affected, because status is no longer pending. His API call returns 409: "already decided by Alice at 14:02."
  4. HumanBob's dashboard shows the existing decision. No second approval is ever recorded — the guard is the state machine itself, not UI politeness.

07

Policy recursion — the bot proposing its own rules

The elegant trick: creating a policy is itself just an action, with type aps.policy.create, flowing through the same engine, queue, and audit trail as everything else. No second approval system exists.

E — Bot proposes a policy

  1. BotHas been asking humans to approve SELECTs all day. Submits action aps.policy.create whose payload is the proposed policy: "db.query matching ^\s*SELECT → allow".
  2. ServiceBy default no policy covers aps.policy.create, so it lands in the human queue like anything else. A policy row is stored as pending_approval.
  3. HumanReviews the rendered rule — what it would allow, what it wouldn't — and approves. Policy becomes active, approved_by set, depth = 1.
  4. BotFrom now on, SELECTs sail through. The audit trail shows the whole story: who approved the rule, and every action the rule later allowed.

One rule is hard-coded: aps.policy.create can never be auto-allowed. The engine rejects any policy whose effect would be allow for policy creation — so every bot-proposed rule passes a human, always, no matter what policies exist. What a bot policy can grant is the right to propose further policies; each proposal records which policy authorized it (authorized_by_policy_id) and its depth — parent depth + 1, humans are depth 0. Creation past the configured limit (default 3) is rejected outright. Revoking a policy in the chain disables its descendants with it.

Why this closes the scope problem. A depth counter limits chain length, not breadth — the scary scenario was a bot-authored rule far broader than intended slipping into effect. With human approval mandatory for every bot policy, breadth is always human-reviewed. Formal "child must be narrower than parent" verification only becomes necessary if this invariant is ever relaxed (see resolved question №4).

Open questions — ranked by how much they'd bend the architecture

№ImpactQuestionCurrent lean
1HighSummary spoofing. The bot writes its own summary. If humans approve from the summary, the bot controls what they think they approved.Queue UI always renders the raw payload; summary is a visually-secondary caption. This is a UI-contract, decided now — no LLM required: the defense is deterministic (show the real payload, pretty-printed per action type). A service-side LLM summarizer could be a later convenience, but never the basis of a decision, since a summarizer can itself be steered by adversarial payload content.
2ResolvedApproved-but-never-executed. Bot gets approval, crashes, retries an hour later. Is the approval still good?Resolved: every request carries a TTL — default 15 minutes, longer only if the bot asked before approval (ttl_seconds, server-capped, shown to the approver). One expires_at deadline bounds both the decision and the execution; an unconsumed approval expires with it.
3DeferredPayload sensitivity. Payloads (and the audit log) will contain real SQL, tokens, customer data. Retention? Redaction? Who can read the audit trail?Deferred — demo scope. Everything is stored as-is, dashboard behind login. Revisit (retention, redaction, audit-read access) before pointing this at any real data.
4ResolvedScope monotonicity for bot policies. Depth limits chain length, not breadth — a bot-authored rule could still be far broader than intended.Resolved: every bot-created policy requires human approval — the engine refuses any policy that would auto-allow aps.policy.create, so no meta-policy can bypass review. Breadth is therefore always human-checked; formal "child narrower than parent" verification is only needed if that invariant is ever relaxed.
5ResolvedKill switch. When the bot misbehaves, what's the one-click "stop everything"?Resolved: two levels, both in v1. Per-bot: disabled_at ⇒ deny-all before any policy runs (POST /v1/bots/:id/disable). Global: auto_allow_suspended in system_settings downgrades every allow to require_approval (PUT /v1/system/auto-allow). Both one click in the dashboard, both audit-logged.
6MedNotifications. Humans won't sit watching a queue; a 15-minute expiry assumes they hear about requests fast.V1 is dashboard-only (plus browser notifications, maybe). Slack/email webhooks are v1.x — same reserved webhook surface as bot callbacks.
7MedPending-flood rate limiting. A looping bot could bury the queue in hundreds of requests.Cap concurrent pendings per bot (e.g. 20); beyond it, submissions get 429. Idempotency keys already stop accidental duplicates.
8MedLLM matcher semantics. Non-deterministic verdicts, per-check cost, and "what did the model see?" auditability.Designed-in: fail-closed on unreachable/uncertain, rationale stored in audit_events.details, only ever narrowing (allowed to say "ask a human", never to override a deny).
9MedMulti-instance / scale-out. The in-process expiry sweeper assumes a single service node.Postgres already supports multiple nodes; when scaling out, the sweeper needs a leader lock (pg_advisory_lock) so only one instance sweeps.
